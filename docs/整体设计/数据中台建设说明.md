# 数据中台建设说明

## 1. 当前阶段定位

当前数据中台的主要目标不是直接建设复杂的经营分析体系，而是先解决数据自动化和可信数据供给问题。

公司主要业务围绕棕榈种植、CPO 生产以及产品销售展开。数据中台建设应优先覆盖从园区现场作业、鲜果收获、工厂加工、库存发货到销售和财务结算的核心业务链路。

核心目标是：

- 将 SAP、Dynamics、园区现场系统和业务 Excel 中的数据自动接入中台。
- 对数据进行清洗、校验、标准化和统一存储。
- 让业务部门可以快速提取正确、可信、可追溯的数据。
- 为后续经营分析、驾驶舱、预警和绩效分析打基础。

因此，当前阶段可以定义为：

> 建设统一、可信、可追溯的数据接入与标准化平台，将 SAP、Dynamics、园区现场系统与业务 Excel 数据自动采集、清洗、校验、入库，并形成可复用的标准数据表，支持各部门快速取数和自主分析。

## 2. 参考方法论与能力框架

结合外部数据中台方法论和公司当前阶段，数据中台不应被理解为单一数据库、单一报表系统或单纯的数据管理工具，而应理解为一套让数据持续流动、持续沉淀、持续复用并服务业务的机制。

对公司当前阶段来说，数据中台建设应重点围绕四类能力展开。

### 2.1 汇聚整合能力

汇聚整合解决的是“数据从哪里来、如何稳定进入中台”的问题。

对应到公司场景，主要包括：

- 接入 SAP 中的工农业相关业务数据。
- 接入 Dynamics 中的财务、采购等业务数据。
- 接入 Long Tech、Long Warehouse、Smart Salary 等园区现场系统数据。
- 接入系统周期不能满足、系统中暂无或需要人工补充的 Excel / 表单数据。
- 建立数据目录、数据源清单、接入批次、文件留痕和基础权限控制。

这一阶段的核心不是先做复杂分析，而是先把多源数据稳定、完整、可追溯地接进来。

### 2.2 提纯加工能力

提纯加工解决的是“接进来的数据如何变成可信数据资产”的问题。

对应到公司场景，主要包括：

- 通过 ODS 保存原始数据和原始文件。
- 通过 DWD 形成标准明细数据。
- 通过 DWS 沉淀高频复用的主题汇总数据。
- 通过主数据和映射表统一公司、园区、小区、地块、工厂、产品、物料、客户、供应商、员工、车辆等编码。
- 通过数据质量规则处理缺漏、不一致、格式错误、重复、逻辑异常和跨系统冲突。
- 通过指标口径表明确字段含义、计算公式、统计周期、责任部门和核对规则。

这一阶段是当前项目的建设重点。只有数据完成提纯加工，后续业务取数和经营分析才有稳定基础。

### 2.3 服务可视化能力

服务可视化解决的是“业务如何方便地使用中台数据”的问题。

对应到公司场景，主要包括：

- 通过 ADS 建立面向业务的标准取数数据集。
- 通过 SQL 视图、Excel 导出、BI 工具或 API 提供统一数据出口。
- 为农业、工厂、库存、发货、财务、采购、薪资等主题提供可复用数据集。
- 控制业务用户优先访问 ADS 或授权视图，避免直接查询 ODS 原始数据。

当前阶段可以先提供标准表、视图和 Excel 导出，后续再逐步扩展到 BI 看板、接口服务和自动化报表。

### 2.4 价值应用能力

价值应用解决的是“数据如何进一步驱动业务行动”的问题。

后续在数据稳定后，可以逐步支撑：

- 经营分析
- 管理驾驶舱
- 指标自动计算
- 异常预警
- 数据质量监控
- 绩效评估
- 成本、产量、库存、发货等专题分析

当前阶段不宜一开始追求完整的高级分析应用，而应先聚焦数据接入、治理、标准化和可信取数。等数据链路跑通后，再按高价值业务场景逐步建设上层应用。

### 2.5 落地原则

结合两篇文章的方法论和公司实际情况，数据中台建设应遵循以下原则：

- 业务牵引：围绕核心工农业指标和高频取数场景反推数据源、表、字段和口径。
- 系统优先：能从系统稳定获取的数据优先走系统，Excel 和手工录入只作为补充和兜底。
- 先小后大：先选 3-5 个高价值场景或核心指标跑通，再逐步扩展。
- 先治理后应用：先解决标准、口径、质量、主数据和血缘追溯，再建设复杂分析。
- 共享复用：DWD、DWS、ADS 应尽量沉淀可复用数据资产，避免每个部门重复取数、重复清洗、重复计算。
- 可追溯：每条数据都应能追溯到来源系统、原始文件、接入批次、处理规则和责任人。
- 持续迭代：数据中台不是一次性项目，而是随业务系统、指标口径和管理要求持续演进的数据能力体系。

## 3. 指标清单的定位

现有指标清单来自业务部门提供的业务表，并经过整理和优化。它可以作为数据中台建设的重要业务输入，但建议定位为：

- 业务数据目录初稿
- 数据供给需求清单
- 数据自动化需求清单
- 指标资产目录初稿

它可以反推三类内容：

- 哪些 Excel、SAP、Dynamics 和现场系统数据必须接入。
- 应沉淀哪些标准事实表和维度表。
- 哪些字段需要统一命名、统一口径、统一编码。

后续如果要升级为正式业务需求，还需要业务部门确认指标是否继续使用、口径是否准确、优先级如何、是否用于报表展示或仅作为基础采集字段。

## 4. 数据来源与接入建议

当前数据来源主要分为四类：

- SAP 数据
- Dynamics 数据
- 园区现场系统数据，例如 Long Tech、Long Warehouse、Smart Salary
- Excel 业务表数据

系统分工建议按当前规划明确为：

- 工农业相关业务数据放在 SAP 中，SAP 作为工农业业务数据的主要系统来源。
- 财务、采购等其他业务数据放在 Dynamics 中，Dynamics 作为非工农业业务数据的主要系统来源。
- Long Tech、Long Warehouse、Smart Salary 等园区现场系统作为现场作业、仓储、薪资等专项业务数据来源。
- Excel 作为现场业务表和过渡性数据来源，后续逐步通过模板化、自动化接入中台。

数据接入优先级应遵循“系统优先、文件补充、手工兜底”的原则：

1. 如果数据可以从 SAP、Dynamics、Long Tech、Long Warehouse、Smart Salary 等系统稳定获取，应优先走系统接入。
2. 如果系统中有数据，但更新周期、字段粒度或接口能力暂时不能满足业务取数需求，可以阶段性使用 Excel 补充。
3. 如果系统中没有该类数据，或属于临时说明、异常原因、业务备注等非结构化信息，可以通过 Excel 或表单进行手工录入。
4. 同一业务事实同时存在系统数据和 Excel 数据时，原则上以系统数据为准，Excel 用于补充、核对或过渡。

### 4.1 SAP 数据接入

SAP 适合作为工农业相关业务数据的主要系统来源，优先承载农业、工厂生产、库存、发货、地磅、分级等与工农业经营过程直接相关的数据。

SAP 适合优先承载：

- 农业生产过程数据，例如采收、施肥、养护、机械、降雨、出勤等。
- 工厂生产过程数据，例如收果、加工、产量、出油率、损耗、停机等。
- 工厂库存与发货过程数据，例如 CPO、PK、CPKO、PKE 等产品库存和发运记录。
- 地磅、分级、车辆、来源地、园区、地块等现场经营数据。
- 工农业相关的组织、工厂、园区、地块、产品等主数据。

SAP 接入方式建议优先级：

1. SAP API / OData / RFC 等标准接口。
2. CDS View 或数据库视图。
3. 定时导出文件，作为过渡方案。
4. 直接读取底层表，需谨慎，并由 SAP 顾问确认业务口径。

### 4.2 Dynamics 数据接入

后续如果引入 Dynamics，建议将其作为财务、采购等非工农业业务的主要系统接入源，而不是只把它看作普通文件来源。

Dynamics 适合优先承载：

- 财务科目、会计期间、凭证、费用、成本等财务数据。
- 采购申请、采购订单、供应商、收货、付款等采购数据。
- 客户、合同、销售订单、应收等销售相关数据。
- 与组织、部门、成本中心、项目、币种、税务相关的主数据。

Dynamics 接入方式建议优先级：

1. 标准 API / OData 接口。
2. 数据实体导出或数据管理框架。
3. 定时同步到中间库或文件区。
4. 报表导出文件，作为过渡方案。

需要重点确认：

- 工农业业务数据以 SAP 为准，财务、采购等其他业务数据以 Dynamics 为准。
- SAP 与 Dynamics 的主数据编码如何映射，例如公司、部门、成本中心、产品、物料、供应商、客户。
- 如果同一业务对象在 SAP 和 Dynamics 中都存在，应按业务域确定权威来源，避免两个系统同时作为同一指标的计算依据。
- 历史 SAP 数据与新 Dynamics 数据如何衔接。

### 4.3 园区现场系统接入

Long Tech、Long Warehouse、Smart Salary 等系统是园区现场实际使用的专项系统，应作为中台的重要数据来源之一。

现场系统适合优先承载：

- Long Tech：园区现场作业、种植过程、采收或生产相关的业务记录，具体字段以后续系统调研为准。
- Long Warehouse：园区或工厂仓储、出入库、库存、物资流转等数据。
- Smart Salary：现场员工、出勤、计件、工资、奖金、扣款等薪资相关数据。

现场系统接入建议：

1. 优先确认每个系统是否提供 API、数据库视图或标准导出。
2. 如果暂时只能导出 Excel/CSV，则先按文件接入方式进入 ODS。
3. 识别系统中的标准编码，例如员工、园区、小区、地块、物料、仓库、车辆等。
4. 与 SAP、Dynamics 的主数据进行映射，避免同一对象在多个系统中编码不一致。
5. 对现场系统与 Excel 报表重叠的数据建立权威来源规则，优先使用系统数据，Excel 作为补充或过渡。

需要重点确认：

- 每个现场系统覆盖的业务范围和数据表清单。
- 是否能稳定自动导出或接口同步。
- 与 SAP / Dynamics 是否存在重复业务数据。
- 哪些 Excel 报表可以被现场系统数据替代。

### 4.4 Excel 数据接入

短期不建议废除 Excel，而是采用“标准模板 + 自动采集 + 自动校验 + 入库”的方式。Excel 的定位不是优先数据源，而是系统周期不能满足、系统中暂时没有数据、或需要人工补充说明时的补充来源。

Excel 接入建议：

- 固定模板，减少随意改列名、合并单元格、跨行标题等情况。
- 每张表带有报表名称、区域、部门、统计日期、填报人、模板版本等信息。
- 上传后先进入暂存区，校验通过后再进入标准层。
- 保留原始 Excel 文件，方便追溯。
- PostgreSQL 中存储文件路径、批次号、上传人、校验结果等元信息。

## 5. 数据治理建议

数据存在缺漏、不一致、输入错误时，治理不能只依靠后期人工修正，而应嵌入数据接入流程。

### 5.1 常见问题类型

- 缺漏：日期、园区、产品、重量等必填项为空。
- 不一致：同一园区、产品、客户、车辆存在多种写法。
- 输入错误：日期格式错误、数字填成文本、单位不一致。
- 重复数据：同一报表、同一日期、同一区域重复上传。
- 口径冲突：不同表中同一指标定义不同。
- 逻辑异常：扣重后重量大于扣重前、库存为负、完成率异常。
- 来源冲突：SAP、Dynamics、现场系统、Excel 对同一业务事实的数据不一致。

### 5.2 校验规则

建议将质量规则分为三类：

- 强校验：不通过不能入库，例如日期为空、数值格式错误。
- 弱校验：可以入库但标记异常，例如出油率超出合理范围。
- 对账校验：入库后定期检查，例如 SAP / Dynamics / 现场系统与 Excel 数据差异。

常见校验包括：

- 必填校验
- 格式校验
- 主数据匹配校验
- 合理范围校验
- 业务逻辑校验
- 唯一性校验
- 跨表对账校验

### 5.3 主数据治理

不一致问题大多来自没有统一编码。建议建立统一主数据和别名映射。

重点主数据包括：

- 区域
- 公司
- 园区
- 小区
- 地块
- 工厂
- 产品
- 物料
- 客户
- 供应商
- 车辆

Excel 中的填报文本不应直接作为分析维度，应先映射为标准编码。

示例：

| 来源名称 | 标准编码 |
| --- | --- |
| WPP | EST_WPP |
| PT WPP | EST_WPP |
| W.P.P | EST_WPP |

### 5.4 治理闭环

建议形成以下闭环：

1. Excel 上传、SAP 抽取、Dynamics 抽取或现场系统抽取。
2. 系统自动校验。
3. 生成错误清单。
4. 反馈给填报人或数据责任人。
5. 修改后重新上传，或由授权人员修正。
6. 记录修正原因、修正人、修正时间。
7. 数据通过后进入标准层。

## 6. PostgreSQL 选型建议

第一阶段使用 PostgreSQL 是可行且合适的。

PostgreSQL 可以作为第一阶段的：

> 数据中台核心数据库 + 轻量数据仓库

适合承载：

- Excel 解析后的结构化数据
- SAP / Dynamics / 现场系统抽取数据
- ODS / DWD / DWS / ADS 分层表
- 主数据、映射表、指标口径表
- 数据质量校验结果
- SQL 查询、视图、物化视图
- Power BI、FineBI、Superset、Metabase、Excel 等取数工具

建议搭配：

- 数据库：PostgreSQL
- 原始文件：本地目录、NAS、MinIO、SharePoint 或对象存储
- 数据接入：Python，或结合 SAP / Dynamics / 现场系统标准接口工具
- 调度：Windows Task Scheduler / cron，后续可引入 Airflow 或 Prefect
- 数据建模：SQL，后续可引入 dbt
- 业务取数：BI 工具、Excel 直连、SQL 视图或导出文件

如果未来出现几十 TB 数据量、高并发复杂分析、实时流数据或 IoT 秒级数据，再考虑 ClickHouse、Doris、Greenplum、Trino 或湖仓架构。

## 7. dbt 的定位

dbt，全称 data build tool，是数据建模和转换管理工具。

它不负责从 Excel、SAP、Dynamics 或现场系统接入数据，而是负责数据进入 PostgreSQL 之后，从 ODS 转换到 DWD、DWS、ADS。

数据流示例：

```text
Excel / SAP / Dynamics / 现场系统
  ↓
Python 接入
  ↓
PostgreSQL ODS 原始表
  ↓
dbt 清洗、转换、建模
  ↓
DWD / DWS / ADS 标准表
  ↓
BI / Excel / 业务取数
```

dbt 的价值：

- 管理 SQL 模型。
- 自动处理表之间的依赖关系。
- 做数据质量测试。
- 生成数据文档。
- 支持 Git 版本管理。

dbt 不是第一天必须引入。建议第一阶段先用 Python + PostgreSQL + SQL 跑通流程，等 SQL 脚本变多、表依赖复杂、多人协作增多时再引入。

## 8. 分层存储设计

数据分层的意思是：数据进入中台后，不是全部混在一起，而是按原始程度、清洗程度、汇总程度和使用场景分层保存。

建议采用轻量四层：

```text
SAP / Dynamics / 现场系统 / Excel / 表单
        ↓
ODS：原始接入层
        ↓
DWD：标准明细层
        ↓
DWS：主题汇总层
        ↓
ADS：应用取数层
```

### 8.1 ODS 原始接入层

ODS 用于保存刚接进来的原始数据。

主要目标：

- 接住数据。
- 保留原貌。
- 支持追溯。
- 支持重跑清洗规则。

ODS 中保存：

- SAP / Dynamics / 现场系统原始抽取数据
- Excel 原始解析结果
- 文件名
- 上传批次
- 上传人
- 上传时间
- 模板版本
- 数据来源

示例表：

- `ods.excel_fertilizer_raw`
- `ods.sap_inventory_raw`

### 8.2 DWD 标准明细层

DWD 用于保存清洗后的标准业务明细数据。

这一层会做：

- 字段统一命名
- 日期格式统一
- 单位统一
- 主数据编码映射
- 异常数据标记
- 去重
- 合并同类业务数据

示例表：

- `dwd.fertilizer_detail`
- `dwd.weighbridge_detail`
- `dwd.factory_production_detail`

DWD 是后续自动化取数的核心基础。

### 8.3 DWS 主题汇总层

DWS 用于保存按业务主题提前汇总的数据。

适合保存：

- 每日园区施肥汇总
- 每月工厂产量汇总
- 每日库存汇总
- 每月发货汇总
- 每个地块采收汇总

示例表：

- `dws.fertilizer_estate_daily`
- `dws.factory_production_monthly`
- `dws.inventory_daily`

DWS 不必一开始全部建设，可先针对高频查询和常用汇总建设。

### 8.4 ADS 应用取数层

ADS 用于直接面向业务取数、BI、报表或导出。

适合保存：

- 农业日报数据集
- 工厂生产日报数据集
- 库存看板数据集
- 发货统计数据集
- 部门取数宽表

示例表：

- `ads.agriculture_daily_dataset`
- `ads.factory_production_dataset`
- `ads.inventory_dataset`

ADS 的作用是隐藏底层复杂结构，为业务提供开箱即用的数据集。

### 8.5 四层是否都必要

从概念上建议保留四层。

从物理落表上可以简化：

| 层级 | 第一阶段建议 |
| --- | --- |
| ODS | 必须落表 |
| DWD | 必须落表 |
| DWS | 高频汇总才落表，不常用可用视图 |
| ADS | 核心取数场景才建表或视图 |

## 9. 数据库组织方式

第一阶段建议放在同一个 PostgreSQL 实例、同一个业务数据库中，用不同 schema 分层。

示例：

```text
julong_data_platform
├── ods   原始接入层
├── dwd   标准明细层
├── dws   主题汇总层
├── ads   应用取数层
├── dim   维度/主数据层
├── meta  元数据/批次/质量规则
└── audit 审计/日志
```

建 schema 示例：

```sql
create schema ods;
create schema dwd;
create schema dws;
create schema ads;
create schema dim;
create schema meta;
create schema audit;
```

同库不同 schema 的优点：

- 管理简单。
- 跨层 SQL 查询方便。
- 权限可以按 schema 控制。
- 备份恢复简单。
- 不需要复杂跨库同步。
- 适合第一阶段轻量落地。

后续如果数据量、权限隔离、性能压力或计算引擎需求增加，再考虑拆库或引入其他存储。

## 10. dim、meta、audit 三类支撑层

`ods/dwd/dws/ads` 存业务数据，`dim/meta/audit` 支撑业务数据标准化、可追溯和可管理。

### 10.1 dim 维度/主数据层

`dim` 存放业务分析中反复使用的标准对象。

包括：

- 区域
- 公司
- 园区
- 小区
- 地块
- 工厂
- 产品
- 物料
- 客户
- 供应商
- 车辆
- 员工

示例表：

- `dim.estate`
- `dim.product`
- `dim.factory`
- `dim.estate_mapping`

它主要解决名称不一致、编码不统一的问题。

### 10.2 meta 元数据/批次/规则层

`meta` 存放管理数据的数据。

包括：

- 数据接入批次
- 文件上传记录
- Excel 模板信息
- 字段映射规则
- 数据质量规则
- 指标定义
- 表字段说明
- 调度任务配置

示例表：

- `meta.ingestion_batch`
- `meta.data_quality_rule`
- `meta.metric_definition`
- `meta.excel_template`

它主要解决接入、规则、指标和模板可配置、可管理的问题。

### 10.3 audit 审计/日志层

`audit` 存放系统运行和数据处理过程中的日志。

包括：

- 数据导入日志
- 数据校验结果
- 数据修改记录
- 异常数据记录
- 用户查询日志
- 数据导出日志
- 权限访问日志

示例表：

- `audit.data_quality_result`
- `audit.data_change_log`
- `audit.ingestion_log`

它主要解决可追溯、可排查、可审计的问题。

## 11. 批次字段 source_batch_id

`source_batch_id` 表示某条数据来自哪一次数据接入批次。

例如某业务人员上传一份 Excel：

```text
batch_id = B202605210001
文件名 = 施肥日报_20260520.xlsx
上传人 = 张三
上传时间 = 2026-05-21 09:30
数据来源 = Excel
```

这份 Excel 解析出的所有记录都带上：

```text
source_batch_id = B202605210001
```

这样以后某条数据有问题时，可以追溯到：

- 来自哪个 Excel 文件
- 谁上传的
- 什么时间上传的
- 用的哪个模板版本
- 校验是否通过
- 后续是否被修正过

建议每条接入数据都保留类似字段：

- `source_system`
- `source_file_name`
- `source_batch_id`
- `ingested_at`
- `data_quality_status`
- `created_at`
- `updated_at`

## 12. 第一阶段落地建议

第一阶段建议不要追求大而全，而是先围绕核心报表和高频取数场景跑通闭环。

建议优先事项：

1. 确定第一批核心 Excel 报表、SAP 数据对象、Dynamics 数据对象和现场系统数据对象。
2. 建立 PostgreSQL 数据库和 schema。
3. 建立主数据和映射表。
4. 建立 Excel 文件接入、解析、批次记录机制。
5. 建立 ODS 原始表。
6. 建立 DWD 标准明细表。
7. 建立数据质量校验和错误反馈机制。
8. 针对高频取数场景建立 ADS 数据集。
9. 后续逐步补充 DWS 汇总层和 dbt 工程化管理。

第一阶段成功标准：

- 原始数据可追溯。
- 错误数据可发现。
- 标准数据可使用。
- 异常数据可反馈。
- 关键主数据统一编码。
- 核心报表可以自动入库。
- 业务部门可以从 ADS 或标准视图快速提取可信数据。
