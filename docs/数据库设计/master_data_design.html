设计决策说明

① 工厂直接挂在公司下，不挂在园区下

工厂（POM/KCP/精炼）是独立法人实体，可能同时接收多个园区的 FFB。如果挂在园区下，会造成逻辑错误——例如七园的工厂也处理八园的果。正确结构：公司 → 工厂，DWD 事实表里用 site（来源园区）和 factory_site（工厂所在地）分别记录。

② company_code 冗余向下复制到每一层

dim_division 和 dim_blok 都直接存一份 company_code。这是数据仓库的标准做法（反范式化），避免每次查询都 JOIN 到顶层公司表。例如查"某公司所有小区的出勤率"，直接 WHERE company_code = 'GIN'，无需三层 JOIN。

③ 地块静态信息放 DIM，动态信息放 DWD

放 dim_blok（变化慢，每季度/年才改）：luas_ha、planting_year、maturity_stage、palm_count、sph、soil_type
放 dwd_agri_harvest_rotation_monthly（每月变化）：current_days_since_harvest、mtd_harvested_ha、harvest_round_count

④ DWD 事实表存字符串代码，不存外键

事实表（如 dwd_agri_harvest_daily）里的 site、division、blok 存字符串代码，而非整型外键。这是 Kimball 维度建模的标准做法——数仓查询走 JOIN，不走数据库 FK 约束，保持事实表的独立性和加载灵活性。

⑤ 数据来源字段（source）

每张 DIM 表都有 source 字段，值为 manual（手动维护）或 sync（从 SAP/外部数据源同步）。同步来的记录通过 external_id 与外部系统对应，更新时以 code 为主键做 upsert，不重复创建。

DIM 层 vs DWD 层：哪些信息放哪里

DIM 主数据层 — 静态描述信息
  • ✓ 地块面积 luas_ha
  • ✓ 种植年份 planting_year
  • ✓ 成熟阶段 maturity_stage
  • ✓ 棕榈棵数 palm_count
  • ✓ 株距密度 sph
  • ✓ 工厂产能 capacity_ton_per_hour
  • ✓ 小区管理面积 managed_area_ha
  • — 这些信息每季度/年才会变化
DWD 事实层 — 动态每日/每月数据
  • ✓ 当日铲果产量 actual_kg
  • ✓ 月累计产量 mtd_actual_kg
  • ✓ 当前距上次铲果天数
  • ✓ 本月累计铲果面积
  • ✓ 当日出勤人数
  • ✓ 当日降雨量
  • ✓ 当日施肥完成量
  • — 这些信息每天/每月更新
